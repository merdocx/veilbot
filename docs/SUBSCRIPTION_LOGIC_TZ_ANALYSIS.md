## Проверка текстов уведомлений

### Текущие тексты уведомлений

**1. Уведомление о покупке** (`is_purchase=True`):
```
✅ *Подписка успешно создана!*

🔗 *Ссылка (коснитесь, чтобы скопировать):*
`{subscription_url}`

⏳ *Срок действия:* {format_duration(tariff['duration_sec'])}

💡 *Как использовать:*
1. Откройте приложение V2Ray
2. Нажмите "+" → "Импорт подписки"
3. Вставьте ссылку выше
4. Все серверы будут добавлены автоматически
```

**2. Уведомление о продлении** (`is_purchase=False`):
```
✅ *Подписка успешно продлена!*

🔗 *Ссылка (коснитесь, чтобы скопировать):*
`{subscription_url}`

⏳ *Добавлено времени:* {format_duration(tariff['duration_sec'])}
📅 *Новый срок действия:* до {datetime.fromtimestamp(new_expires_at).strftime('%Y-%m-%d %H:%M:%S')}

💡 Подписка автоматически обновится в вашем приложении
```

### Проблема в новой логике

**В новой логике нет разделения на покупку/продление**, поэтому нужно определить:

1. **Когда показывать текст о покупке?**
   - Если это первый платеж для подписки?
   - Но как определить "первый платеж", если мы всегда пересчитываем?

2. **Когда показывать текст о продлении?**
   - Если это не первый платеж?
   - Но срок действия теперь рассчитывается от всех платежей, а не добавляется к текущему

### Рекомендации по тексту уведомлений

**Вариант 1: Универсальное уведомление (рекомендуется)**
```

🔗 *Ссылка (коснитесь, чтобы скопировать):*

💡 Подписка автоматически обновится в вашем приложении
```

**Вариант 2: Различение по количеству платежей**
```
✅ *Подписка {успешно создана|обновлена}!*

🔗 *Ссылка (коснитесь, чтобы скопировать):*
`{subscription_url}`

{Если первый платеж:}
⏳ *Срок действия:* {format_duration(total_duration_sec)}

{Если не первый платеж:}
⏳ *Добавлено времени:* {format_duration(current_payment_duration_sec)}
📅 *Новый срок действия:* до {datetime.fromtimestamp(new_expires_at)}

💡 {Если первый платеж: инструкция по использованию | Если не первый: "Подписка автоматически обновится"}
```

**Вариант 3: Детализация по платежу**
```
✅ *Платеж обработан успешно!*

🔗 *Ссылка (коснитесь, чтобы скопировать):*
`{subscription_url}`

📦 *Оплачено:* {tariff['name']} - {format_duration(tariff['duration_sec'])}
📅 *Срок действия подписки:* до {datetime.fromtimestamp(new_expires_at)}

💡 Подписка автоматически обновится в вашем приложении
```

### ❌ НЕ учтено в ТЗ (требует добавления)

#### 1. Тексты уведомлений
**Проблема:** В новой логике нет разделения покупка/продление, но тексты уведомлений используют это разделение.

**Что добавить в ТЗ:**
- Раздел "Формирование текста уведомлений"
- Выбор варианта текста (см. рекомендации выше)
- Логика определения: первый платеж или нет (для варианта 2/3)

#### 2. Отправка запасных Outline ключей
**Текущее поведение:** Outline ключи отправляются только при `is_purchase=True`.

**Вопрос:** Отправлять ли Outline ключи при каждом платеже или только при первом?

**Рекомендация:** Только при первом платеже (чтобы не спамить пользователей).

**Что добавить в ТЗ:**
- Раздел "Отправка запасных Outline ключей"
- Условие: отправлять только если `payments_count == 1`
- Или: отправлять только один раз (по флагу в подписке)

#### 3. Обновление subscription_id в платеже
**Текущее поведение:** `update_subscription_id` вызывается с retry механизмом.

**Вопрос:** Нужно ли это в новой логике?

**Рекомендация:** Да, сохранить retry механизм, так как связь payment ↔ subscription критична.

**Что добавить в ТЗ:**
- Упомянуть в разделе "Детали реализации"
- Сохранить retry механизм для `update_subscription_id`

#### 4. Флаг purchase_notification_sent в subscriptions
**Текущее поведение:** Флаг `purchase_notification_sent` используется для определения покупки vs продления.

**Вопрос:** Нужен ли этот флаг в новой логике?

**Рекомендация:** 
- **Вариант А**: Заменить на `first_notification_sent` (отправлено ли первое уведомление)
- **Вариант Б**: Использовать поле `notification_sent` в таблице `payments` (по одному флагу на платеж)

**Что добавить в ТЗ:**
- Раздел "Управление флагами уведомлений"
- Выбор: использовать `purchase_notification_sent` или `notification_sent` в payments
- Миграция флагов при переходе

#### 5. Обработка VIP подписок
**Текущее поведение:** VIP подписки имеют `expires_at = 4102434000` (01.01.2100).

**Вопрос:** Нужно ли пересчитывать expires_at для VIP подписок?

**Рекомендация:** Нет, VIP подписки должны игнорироваться при пересчете (оставаться с фиксированным сроком).

**Что добавить в ТЗ:**
- Условие в `calculate_subscription_expires_at`: если `is_vip = True`, вернуть `VIP_EXPIRES_AT`
- Проверка VIP статуса перед расчетом

#### 6. Сброс трафика при обновлении подписки
**Текущее поведение:** При продлении вызывается `reset_subscription_traffic()`.

**Вопрос:** Нужно ли сбрасывать трафик при каждом платеже?

**Рекомендация:** Нет, сбрасывать только при явном продлении (когда expires_at увеличивается), а не при пересчете.

**Что добавить в ТЗ:**
- Раздел "Сброс трафика"
- Условие: сбрасывать только если `new_expires_at > old_expires_at`
- Или: не сбрасывать вообще (так как это пересчет, а не продление)

#### 7. Создание ключей при первом платеже
**Текущее поведение:** При создании подписки создаются ключи на всех серверах.

**Вопрос:** Создаются ли ключи при `get_or_create_subscription`, если подписка уже существует?

**Рекомендация:** 
- Если подписка существует → не создаем ключи (они уже есть)
- Если создаем новую подписку → создаем ключи

**Что добавить в ТЗ:**
- Раздел "Создание ключей"
- Условие: создавать ключи только если подписка создана (не существовала)
- Логика: `if subscription.created_at == now: create_keys()`

#### 8. Обработка платежей с разными тарифами
**Текущее поведение:** Продление использует `tariff['duration_sec']` текущего платежа.

**В новой логике:** Суммируются длительности всех платежей, независимо от тарифа.

**Вопрос:** Это правильное поведение? ✅ Да, это именно то, что нужно.

**Что добавить в ТЗ:**
- Упомянуть в разделе "Формула расчета expires_at"
- Пример: если платеж 1 (30 дней) + платеж 2 (30 дней) = 60 дней

#### 9. Обновление tariff_id в подписке
**Текущее поведение:** При продлении обновляется `tariff_id` подписки на тариф текущего платежа.

**Вопрос:** Какой `tariff_id` использовать, если платежи с разными тарифами?

**Рекомендация:** Использовать `tariff_id` последнего платежа (как сейчас).

**Что добавить в ТЗ:**
- Раздел "Обновление tariff_id"
- Логика: `subscription.tariff_id = last_payment.tariff_id`

#### 10. Обновление traffic_limit_mb
**Текущее поведение:** При продлении обновляется `traffic_limit_mb` из тарифа.

**Вопрос:** Какой `traffic_limit_mb` использовать при пересчете?

**Рекомендация:** Использовать `traffic_limit_mb` последнего платежа (как сейчас).

**Что добавить в ТЗ:**
- Раздел "Обновление traffic_limit_mb"
- Логика: `subscription.traffic_limit_mb = last_payment.tariff.traffic_limit_mb`
