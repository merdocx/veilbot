# Анализ упрощенной логики определения покупки vs продления

## Предлагаемая логика

**Простое правило:**
1. Если на момент получения `success` payment **нет подписки** (активной или с grace period) → **ПОКУПКА**
2. Если на момент получения `success` payment **есть подписка** (активная или с grace period, любая, включая бесплатную) → **ПРОДЛЕНИЕ**

## Сравнение с текущей логикой

### Текущая логика (сложная)
- ✅ 28+ проверок
- ✅ Проверка возраста подписки (< 1 час, < 30 минут)
- ✅ Проверка соответствия expires_at ожидаемому
- ✅ Проверка других completed платежей
- ✅ Проверка флага purchase_notification_sent
- ✅ Специальная обработка бесплатных подписок
- ✅ Разные пути для retry vs нового платежа

### Упрощенная логика (предлагаемая)
- ✅ 1 проверка: есть ли активная подписка (с grace period)
- ✅ Нет проверок возраста подписки
- ✅ Нет проверки соответствия expires_at
- ✅ Нет проверки других платежей
- ✅ Нет проверки флагов уведомлений
- ✅ Единая обработка всех подписок (бесплатных и платных)

## Анализ надежности

### ✅ Преимущества упрощенной логики

1. **Простота**
   - Одна проверка вместо 28+
   - Понятная логика: есть подписка → продление, нет → покупка
   - Легко отлаживать и поддерживать

2. **Идемпотентность**
   - Повторная обработка одного платежа дает тот же результат
   - Если платеж уже `completed` - пропускаем (защита в начале функции)

3. **Правильность для большинства случаев**
   - Первый платеж → покупка ✓
   - Второй платеж → продление ✓
   - Продление бесплатной подписки → продление ✓

4. **Меньше граничных случаев**
   - Нет неоднозначности "59 минут vs 61 минута"
   - Нет зависимости от флагов уведомлений
   - Нет проверки других платежей

### ⚠️ Потенциальные проблемы

#### Проблема 1: Race condition при создании подписки

**Сценарий:**
```
Процесс A: Проверяет отсутствие подписки → создает подписку
Процесс B: Одновременно проверяет отсутствие подписки → пытается создать подписку
```

**Текущая защита:**
- Атомарное создание с проверкой наличия в транзакции (BEGIN IMMEDIATE)
- Если подписка создана другим процессом - используется существующая

**В упрощенной логике:**
- ✅ Та же защита (атомарное создание сохраняется)
- ✅ Если процесс B видит подписку, созданную процессом A - это продление (правильно)

**Вывод:** ✅ **Нет проблемы** - текущая защита от race conditions достаточна

---

#### Проблема 2: Retry обработки первого платежа

**Сценарий:**
1. Первый платеж обрабатывается → создается подписка
2. Webhook retry → платеж обрабатывается повторно
3. На момент retry подписка уже существует

**Текущая логика:**
- Проверяет статус `completed` → пропускает (idempotent)
- Или проверяет `purchase_notification_sent` → не продлевает, только отправляет уведомление

**В упрощенной логике:**
- ✅ Проверка статуса `completed` в начале функции → пропускает (idempotent)
- ⚠️ Если статус еще `paid` (не обработан) → будет определено как **продление**
- ⚠️ Подписка будет продлена повторно (добавлено еще 30 дней)

**Проблема:**
- При retry первого платежа подписка будет продлена дважды:
  - Первая обработка: создана подписка на 30 дней
  - Retry: продлевается еще на 30 дней = 60 дней вместо 30

**Решение:**
- ✅ **Проверка статуса `completed` в начале функции** (уже есть)
- ✅ Если `status = 'completed'` → пропускаем обработку (idempotent)
- ⚠️ Если статус еще `paid` (что редко, но возможно) → может быть двойное продление

**Альтернативное решение:**
- Проверять `subscription_id` в платеже
- Если `payment.subscription_id != None` → не продлевать, только отправить уведомление
- Если `payment.subscription_id == None` → обычная логика (покупка/продление)

**Вывод:** ⚠️ **Потенциальная проблема** - нужна дополнительная защита (проверка `subscription_id` или статуса `completed`)

#### Проблема 3: Обработка параллельных платежей

**Сценарий:**
1. Пользователь оплатил 2 платежа одновременно
2. Оба платежа обрабатываются параллельно
3. Оба проверяют отсутствие подписки → оба создают подписку

**Текущая защита:**
- Атомарное создание с транзакцией
- Один процесс создает подписку, второй использует существующую

**В упрощенной логике:**
- ✅ Та же защита (атомарное создание)
- ⚠️ Второй платеж видит подписку → продлевает (правильно)
- ⚠️ Но может быть race condition: оба процесса продлевают одновременно

**Текущая защита при продлении:**
- Атомарное обновление `expires_at` через SQL (`extend_subscription_by_duration_async`)
- Вычисление нового `expires_at` происходит в SQL, не в Python

**Пример:**
```sql
UPDATE subscriptions 
SET expires_at = expires_at + ?
WHERE id = ?
```

**В упрощенной логике:**
- ✅ Та же защита (атомарное SQL-обновление)
- ✅ Два платежа по 30 дней параллельно → подписка продлится на 60 дней (правильно)

**Вывод:** ✅ **Нет проблемы** - атомарное обновление `expires_at` защищает от race conditions

#### Проблема 4: Подписка создана ДО платежа (бесплатная или автоматическая)

**Сценарий:**
1. Пользователю создана бесплатная подписка (например, при регистрации)
2. Пользователь оплачивает первый платный платеж
3. На момент обработки платежа подписка уже существует

**Текущая логика:**
- Проверяет `has_active_free_subscription` → всегда продление

**В упрощенной логике:**
- Есть подписка (любая) → продление ✓
- ✅ Это правильно - платный платеж продлевает бесплатную подписку

**Вывод:** ✅ **Нет проблемы** - упрощенная логика правильно обрабатывает этот случай

---

#### Проблема 5: Подписка создана автоматически ДО платежа (не бесплатная)

**Сценарий:**
1. Подписка создана автоматически (например, другим процессом) ДО обработки платежа
2. Платеж обрабатывается → видит существующую подписку
3. По упрощенной логике → продление

**Проблема:**
- Если подписка создана с неправильным сроком (например, на 90 дней вместо 30)
- Платеж продлевает её → становится еще больше (90 + 30 = 120 дней)

**Пример (подписка #272):**
- Подписка создана на 90 дней (ошибка)
- Первый платеж обрабатывается → продлевает → 120 дней (еще больше ошибка)
- Второй платеж → продлевает → 150 дней

**В текущей логике:**
- Проверка `is_very_recent AND expires_at_matches_expected` может определить это как покупку
- Но это не сработало для подписки #272 (создана за 14 минут до платежа)

**В упрощенной логике:**
- ❌ Нет проверки неправильного `expires_at`
- ❌ Ошибка будет накапливаться при каждом платеже

**Решение:**
- ✅ Использовать **пересчет `expires_at` на основе всех платежей** (как в новом ТЗ)
- ✅ Не продлевать, а пересчитывать: `expires_at = first_payment + sum(all_payments_duration)`

**Вывод:** ⚠️ **Проблема** - но решается переходом на пересчет вместо продления

#### Проблема 6: Двойное продление при retry первого платежа

**Детальный сценарий:**
1. Первый платеж обрабатывается:
   - Подписки нет → создается подписка: `expires_at = now + 30 дней`
   - Платеж помечается как `completed`
   - Уведомление отправлено

2. Webhook retry (платеж еще `paid`, статус не обновился):
   - Проверка `completed` → статус еще `paid` (не обновлен)
   - Проверка подписки → подписка есть → **продление**
   - Продлевает подписку: `expires_at = expires_at + 30 дней` = 60 дней вместо 30

**Текущая защита:**
- Проверка `purchase_notification_sent` → если отправлено, не продлевает
- Или проверка других completed платежей

**В упрощенной логике:**
- ❌ Нет проверки `purchase_notification_sent`
- ❌ Нет проверки других платежей
- ⚠️ Может продлить подписку дважды

**Решение:**
- ✅ **Проверка `subscription_id` в платеже:**
  ```python
  if payment.subscription_id is not None:
      # Платеж уже связан с подпиской → не продлевать
      # Только отправить уведомление, если не отправлено
      return send_notification_only()
  ```
- ✅ **Или проверка статуса `completed` в начале (должна быть достаточно надежной)**
- ✅ **Или проверка других completed платежей с тем же `payment_id`**

**Вывод:** ⚠️ **Проблема** - нужна защита от retry (проверка `subscription_id` или статуса)
