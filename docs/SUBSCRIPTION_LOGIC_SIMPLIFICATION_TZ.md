# Техническое задание: Упрощение логики обработки подписок

## Проблема

Текущая система имеет сложную логику разделения процессов **покупки** и **продления** подписок, что приводит к:

1. **Ошибкам в расчете сроков** (пример: подписка #272 была создана на 90 дней вместо 60)
2. **Сложности поддержки** - более 28 проверок для определения типа операции
3. **Race conditions** - риск ошибок при параллельной обработке платежей
4. **Трудностям отладки** - множество условных ветвлений

## Цель

Упростить логику обработки подписок, убрав разделение на покупку и продление, и всегда пересчитывая срок действия на основе **всех платежей**.

## Принцип новой логики

**Единая функция обработки платежа:**
- Всегда пересчитываем `expires_at` на основе **всех completed платежей** для подписки
- Если подписки нет - создаем её автоматически
- Один путь выполнения, без ветвления покупка/продление

## Архитектура решения

### 1. Основная функция обработки

```python
async def process_subscription_payment(payment_id: str) -> Tuple[bool, Optional[str]]:
    """
    Универсальная функция обработки платежа подписки.
    Всегда пересчитывает expires_at на основе всех платежей.
    
    Алгоритм:
    1. Проверка статуса платежа (должен быть 'paid')
    2. Получение или создание подписки (idempotent)
    3. Сбор всех completed платежей для подписки
    4. Расчет правильного expires_at
    5. Атомарное обновление expires_at (только если изменился)
    6. Отправка уведомления (один раз на платеж)
    """
```

### 2. Получение или создание подписки

```python
async def get_or_create_subscription(user_id: int, payment: Payment) -> Subscription:
    """
    Получить существующую подписку или создать новую.
    
    Логика:
    - Ищем активную подписку пользователя (is_active = 1)
    - Если нет - создаем новую подписку с уникальным токеном
    - Возвращаем существующую или созданную подписку
    """
```

**Выбранное решение: Вариант А** - создание подписки автоматически при первом платеже.

### 3. Расчет expires_at

```python
def calculate_subscription_expires_at(
    payments: List[Payment], 
    subscription_created_at: int,
    user_id: int
) -> int:
    """
    Рассчитать правильный срок действия подписки на основе платежей.
    
    Формула:
    - Базовая дата = max(первый_платеж, created_at)
    - Суммарная длительность = sum(duration_sec всех платежей)
    - Реферальные бонусы = 30 дней × количество рефералов
    - expires_at = базовая_дата + суммарная_длительность + бонусы
    
    Args:
        payments: Список всех completed платежей для подписки
        subscription_created_at: Timestamp создания подписки
        user_id: ID пользователя (для расчета реферальных бонусов)
    
    Returns:
        Timestamp срока действия подписки
    """
```

**Выбранное решение:** Реферальные бонусы включаются в расчет при каждом обновлении.

### 4. Атомарное обновление

```python
async def recalculate_and_update_subscription_expires_at(
    subscription_id: int,
    payments: List[Payment],
    subscription_created_at: int,
    user_id: int
) -> int:
    """
    Атомарно пересчитать и обновить expires_at подписки.
    
    Использует SQL-транзакцию для предотвращения race conditions.
    Возвращает новое значение expires_at.
    """
```

**Выбранное решение: Вариант А** - использование транзакций и блокировок для предотвращения race conditions.

### 5. Обработка платежей без subscription_id

**Выбранное решение:** Обратная совместимость **не нужна**. Все новые платежи должны иметь `subscription_id`.

**Требуется миграция:**
- Связать старые платежи с подписками (если возможно)
- Или удалить их из расчета (если связь невозможно определить)

## Детали реализации

### Формула расчета expires_at

```python
# Шаг 1: Определяем базовую дату
first_payment_date = min(p.created_at for p in payments)
base_date = max(first_payment_date, subscription_created_at)

# Шаг 2: Суммируем длительности всех платежей
total_duration_sec = sum(
    get_tariff_duration(p.tariff_id) 
    for p in payments 
    if p.status == 'completed'
)

# Шаг 3: Рассчитываем реферальные бонусы
referral_bonuses_sec = calculate_referral_bonuses(user_id, base_date + total_duration_sec)
# Реферальный бонус: 30 дней за каждого реферала, который оплатил

# Шаг 4: Финальный расчет
expires_at = base_date + total_duration_sec + referral_bonuses_sec
```

### Защита от дублирования уведомлений

**Выбранное решение:** Использовать поле `notification_sent` в таблице `payments` (по одному флагу на платеж).

```python
# При отправке уведомления устанавливаем флаг notification_sent = 1 в платеже
# Проверяем флаг перед отправкой
# Если флаг установлен - пропускаем отправку
```

**Альтернатива:** Использовать `purchase_notification_sent` в subscriptions (старый подход), но это менее гибко для множественных платежей.

### Формирование текста уведомлений

**Выбранное решение: Вариант 1** - Универсальное уведомление для всех платежей.

**Текст уведомления:**
```
✅ *Подписка обновлена!*

🔗 *Ссылка (коснитесь, чтобы скопировать):*
`{subscription_url}`

⏳ *Срок действия:* до {datetime.fromtimestamp(new_expires_at).strftime('%Y-%m-%d %H:%M:%S')}

💡 Подписка автоматически обновится в вашем приложении
```

**Логика:**
- Одинаковый текст для всех платежей (первый, второй, и т.д.)
- Показываем новый `expires_at` после пересчета
- Инструкция по использованию не нужна (подписка автоматически обновляется)

**Альтернативы (не используются):**
- Вариант 2: Различение "создана" vs "обновлена" по количеству платежей (избыточно)
- Вариант 3: Детализация по текущему платежу (может запутать пользователя)

### Обработка VIP подписок

**Логика:** VIP подписки имеют фиксированный срок `expires_at = 4102434000` (01.01.2100) и **не пересчитываются**.

```python
VIP_EXPIRES_AT = 4102434000  # 01.01.2100
VIP_TRAFFIC_LIMIT_MB = 0     # Безлимит

if is_vip:
    # VIP подписка - не пересчитываем expires_at
    expires_at = VIP_EXPIRES_AT
    traffic_limit_mb = VIP_TRAFFIC_LIMIT_MB
else:
    # Обычная подписка - пересчитываем на основе платежей
    expires_at = calculate_subscription_expires_at(...)
```

**Проверка VIP статуса:**
- Перед расчетом `expires_at` проверяем `user.is_vip`
- Если VIP - возвращаем `VIP_EXPIRES_AT` без расчета
- Если не VIP - выполняем обычный расчет

### Создание ключей при первом платеже

**Логика:** Ключи создаются только если подписка **создана** (не существовала).

```python
async def get_or_create_subscription(user_id: int, payment: Payment) -> tuple:
    """
    Получить существующую подписку или создать новую.
    
    Returns:
        (subscription, was_created: bool)
    """
    subscription = get_existing_subscription(user_id)
    
    if subscription:
        return subscription, False  # Подписка существовала
    else:
        # Создаем новую подписку
        subscription = create_subscription(...)
        
        # Создаем ключи на всех активных серверах
        create_keys_for_subscription(subscription.id, ...)
        
        return subscription, True  # Подписка создана
```

**Условие создания ключей:**
- Создаем ключи только если `was_created = True`
- Если подписка существовала - ключи уже есть, не создаем

### Отправка запасных Outline ключей

**Выбранное решение:** Отправлять только при **первом платеже** (чтобы не спамить пользователей).

**Логика:**

payments_count = len(get_all_completed_payments_for_subscription(subscription_id))

if payments_count == 1:

    send_outline_backup_keys_notification(...)
else:
    # Последующие платежи - не отправляем (ключи уже есть)
    pass
```

**Альтернатива:** Использовать флаг `outline_keys_notification_sent` в subscriptions (менее гибко).

**Текст уведомления:**

[Список Outline ключей по серверам]
```

### Сброс трафика при обновлении подписки

**Выбранное решение:** Не сбрасывать трафик при пересчете.

**Обоснование:**
- Пересчет `expires_at` - это не продление, а пересчет на основе всех платежей
- Сброс трафика должен происходить только при явном продлении (если таковое понадобится)
- Иначе пользователь может потерять историю использования трафика

**Логика:**
```python
# Не вызываем reset_subscription_traffic() при пересчете
# Трафик остается неизменным
```

**Альтернатива:** Сбрасывать только если `new_expires_at > old_expires_at` (не используется).

### Обновление tariff_id и traffic_limit_mb

**Логика:** Использовать значения из **последнего платежа**.

```python
last_payment = max(payments, key=lambda p: p.created_at)

subscription.tariff_id = last_payment.tariff_id
subscription.traffic_limit_mb = get_tariff(last_payment.tariff_id).traffic_limit_mb
```

**Обоснование:**
- Подписка соответствует текущему активному тарифу
- Если пользователь перешел на другой тариф - подписка обновляется под него
- `traffic_limit_mb` берется из тарифа последнего платежа

### Идемпотентность

Функция должна быть идемпотентной:
- Повторная обработка одного платежа дает тот же результат
- Множественная обработка параллельных платежей безопасна благодаря транзакциям
- Проверка статуса `completed` в начале функции

### Обработка ошибок

1. **Платеж уже обработан** (`status = 'completed'`): пропускаем, возвращаем успех
2. **Подписка не найдена после создания**: повторная попытка с блокировкой
3. **Ошибка расчета expires_at**: логирование и возврат ошибки
4. **Ошибка обновления БД**: откат транзакции, логирование, возврат ошибки

## Миграция данных

### Связывание старых платежей

Для старых платежей без `subscription_id`:

1. **Определить подписку пользователя** по временному окну (±7 дней от created_at)
2. **Обновить subscription_id** в платежах
3. **Пересчитать expires_at** для всех затронутых подписок

```sql
-- Пример SQL для связывания старых платежей
UPDATE payments p
SET subscription_id = (
    SELECT s.id 
    FROM subscriptions s
    WHERE s.user_id = p.user_id
      AND s.is_active = 1
      AND ABS(s.created_at - p.created_at) < (7 * 24 * 3600)
    LIMIT 1
)
WHERE p.subscription_id IS NULL
  AND p.status = 'completed'
  AND p.protocol = 'v2ray'
```

### Валидация миграции

После миграции проверить:
- Все ли платежи связаны с подписками
- Совпадают ли пересчитанные `expires_at` с текущими значениями
- Нет ли платежей без связанной подписки

## Преимущества нового подхода

1. **Простота**: Один путь выполнения вместо множества ветвлений
2. **Правильность**: Всегда считается от всех платежей, не может быть расхождений
3. **Идемпотентность**: Повторная обработка безопасна
4. **Устойчивость к race conditions**: Транзакции и атомарные обновления
5. **Легкость отладки**: Одна точка входа, понятная логика

## Риски и ограничения

1. **Производительность**: При каждом платеже нужно пересчитывать все платежи
   - **Митигация**: Обычно платежей немного (1-10), пересчет быстрый
   - При необходимости можно кэшировать расчет

2. **Реферальные бонусы**: Зависимость от других таблиц (referrals)
   - **Митигация**: Проверка целостности данных при расчете

3. **Миграция старых платежей**: Может быть сложно связать все платежи
   - **Митигация**: Использовать временное окно и логирование неудачных случаев
