# Текущая логика определения покупки vs продления подписки

## Общая схема

Логика находится в `payments/services/subscription_purchase_service.py`, метод `process_subscription_purchase()`.

**Основной принцип:** Система пытается определить, является ли платеж **покупкой** (создание новой подписки) или **продлением** (увеличение срока существующей подписки).

## Константы

```python
VERY_RECENT_THRESHOLD = 3600        # 1 час
RECENT_SUBSCRIPTION_THRESHOLD = 1800 # 30 минут
EXPIRES_AT_MATCH_TOLERANCE = 3600   # 1 час
DEFAULT_GRACE_PERIOD = 86400         # 24 часа
```

## Алгоритм определения (пошагово)

### Шаг 1: Базовая проверка наличия активной подписки

```python
grace_threshold = now - DEFAULT_GRACE_PERIOD  # 24 часа назад

# Ищем активную подписку
SELECT * FROM subscriptions
WHERE user_id = ? 
  AND is_active = 1 
  AND expires_at > grace_threshold  # Подписка активна или истекла менее 24 часов назад
ORDER BY created_at DESC
LIMIT 1
```

**Результат:**
- Если подписка **не найдена** → **ПОКУПКА** (переход к `_create_subscription`)
- Если подписка **найдена** → продолжаем проверки ниже

---

### Шаг 2: Проверка бесплатной подписки

```python
# Если найдена подписка, проверяем, является ли она бесплатной
has_active_free_subscription = (
    subscription_tariff_id == FREE_V2RAY_TARIFF_ID OR
    subscription_price_rub == 0
)
```

**Если `has_active_free_subscription = True`:**
- → **ПРОДЛЕНИЕ** (любая оплата продлевает бесплатную подписку)
- Вызов: `_extend_subscription(..., is_purchase=False)`

**Если `has_active_free_subscription = False`:**
- → продолжаем проверки ниже

---

### Шаг 3: Проверка "очень недавно созданной" подписки

**Условия:**
```python
subscription_age = now - created_at
is_very_recent = subscription_age < VERY_RECENT_THRESHOLD  # < 1 часа

expected_expires_at = created_at + tariff['duration_sec']
expires_at_matches_expected = abs(existing_expires_at - expected_expires_at) < EXPIRES_AT_MATCH_TOLERANCE  # Разница < 1 часа

purchase_notification_not_sent = not purchase_notification_sent
```

**Если `is_very_recent AND expires_at_matches_expected`:**
- Подписка создана менее часа назад И срок соответствует ожидаемому (created_at + duration)
- Это означает, что подписка только что создана

**Дополнительная проверка:**
```python
# Ищем другие completed платежи для этого пользователя
SELECT COUNT(*) FROM payments
WHERE user_id = ? 
  AND tariff_id = ?
  AND status = 'completed'
  AND protocol = 'v2ray'
  AND metadata LIKE '%subscription%'
  AND created_at >= (created_at - VERY_RECENT_THRESHOLD)  # В пределах часа от создания подписки
  AND payment_id != current_payment_id
```

**Если `other_completed_count == 0 OR purchase_notification_not_sent`:**
- → **ПОКУПКА** (подписка только что создана, это первый платеж или retry)
- Вызов: `_send_purchase_notification_for_existing_subscription()`
- **НЕ продлеваем** подписку (она уже создана с правильным сроком)

**Если `other_completed_count > 0 AND purchase_notification_sent = 1`:**
- → продолжаем проверки ниже (это не первый платеж)

---

### Шаг 4: Проверка "недавно созданной" подписки (30 минут)

**Условия:**
```python
is_recent_subscription = subscription_age < RECENT_SUBSCRIPTION_THRESHOLD  # < 30 минут
purchase_notification_not_sent = not purchase_notification_sent
```

**Если `is_recent_subscription AND purchase_notification_not_sent`:**
- Подписка создана менее 30 минут назад И уведомление не отправлено
- Это может быть retry обработки того же платежа

**Дополнительная проверка:**
```python
# Ищем другие completed платежи (та же проверка, что выше)
other_completed_count = COUNT(...)
```

**Если `other_completed_count == 0`:**
- → **ПОКУПКА** (retry того же платежа)
- Вызов: `_send_purchase_notification_for_existing_subscription()`
- **НЕ продлеваем** подписку

**Если `other_completed_count > 0`:**
- → продолжаем проверки ниже

---

### Шаг 5: Финальное решение

**Если дошли до этого шага:**
- Есть активная подписка
- Подписка не бесплатная
- Подписка создана не очень недавно (> 1 часа) ИЛИ срок не соответствует ожидаемому
- ИЛИ подписка создана недавно (> 30 минут) ИЛИ уведомление отправлено
- ИЛИ есть другие completed платежи

**→ ПРОДЛЕНИЕ**
- Вызов: `_extend_subscription(..., is_purchase=False)`
- Подписка продлевается: `expires_at = expires_at + tariff['duration_sec']`

## Итоговая таблица решений

| Условие | Результат | Действие |
|---------|-----------|----------|
| Нет активной подписки | **ПОКУПКА** | `_create_subscription()` |
| Есть бесплатная подписка | **ПРОДЛЕНИЕ** | `_extend_subscription(..., is_purchase=False)` |
| Подписка < 1 часа И срок совпадает И (нет других платежей ИЛИ уведомление не отправлено) | **ПОКУПКА** | `_send_purchase_notification_for_existing_subscription()` (не продлеваем) |
| Подписка < 30 минут И уведомление не отправлено И нет других платежей | **ПОКУПКА** | `_send_purchase_notification_for_existing_subscription()` (не продлеваем) |
| Все остальные случаи (есть активная подписка) | **ПРОДЛЕНИЕ** | `_extend_subscription(..., is_purchase=False)` |

## Детали проверок

### Проверка 1: "Очень недавно созданная" подписка

**Цель:** Определить, была ли подписка только что создана (возможно, другим процессом или автоматически).

**Условия:**
1. `subscription_age < 3600` (менее 1 часа)
2. `abs(existing_expires_at - expected_expires_at) < 3600` (срок совпадает с ожидаемым ± 1 час)
3. `other_completed_count == 0` (нет других completed платежей) ИЛИ `purchase_notification_not_sent` (уведомление не отправлено)

**Если все условия выполнены:**
- Это **покупка** (первый платеж или retry)
- Подписка уже создана с правильным сроком, не нужно продлевать
- Только отправляем уведомление о покупке

### Проверка 2: "Недавно созданная" подписка (30 минут)

**Цель:** Обработать retry обработки того же платежа.

**Условия:**
1. `subscription_age < 1800` (менее 30 минут)
2. `purchase_notification_not_sent` (уведомление не отправлено)
3. `other_completed_count == 0` (нет других completed платежей)

**Если все условия выполнены:**
- Это **покупка** (retry того же платежа)
- Не продлеваем подписку, только отправляем уведомление

### Проверка 3: Grace period (24 часа)

**Цель:** Учесть подписки, которые истекли недавно (в пределах 24 часов).

**Логика:**
- Если подписка истекла менее 24 часов назад, она считается "активной" для продления
- Это позволяет пользователю продлить подписку даже если она только что истекла

## Особые случаи

### Случай 1: Параллельная обработка платежей

**Проблема:** Два процесса одновременно обрабатывают платежи для одного пользователя.

**Решение:**
- Атомарная проверка статуса `completed` в начале функции
- Атомарное обновление `expires_at` через SQL (`extend_subscription_by_duration_async`)
- Транзакции при создании подписки

### Случай 2: Retry обработки платежа

**Проблема:** Платеж обрабатывается повторно (webhook retry, ручной retry).

**Решение:**
- Проверка статуса `completed` в начале функции
- Проверка `purchase_notification_sent` для определения, было ли отправлено уведомление
- Если уведомление отправлено - пропускаем обработку

### Случай 3: Подписка создана, но уведомление не отправлено

**Проблема:** Подписка была создана (возможно другим процессом), но уведомление не отправлено.

**Решение:**
- Проверка `purchase_notification_not_sent`
- Если подписка недавно создана - отправляем уведомление о покупке, не продлеваем

### Случай 4: Ручная установка срока (VIP, админка)

**Проблема:** Срок подписки установлен вручную (VIP = 01.01.2100, или через админку).

**Решение:**
```python
MANUAL_EXPIRY_THRESHOLD = 4102434000  # 01.01.2100
is_manually_set = (
    existing_expires_at >= MANUAL_EXPIRY_THRESHOLD OR
    (existing_expires_at > now AND (existing_expires_at - now) > (5 * 365 * 24 * 3600))
)

if is_manually_set:
    # Не изменяем expires_at, только обновляем tariff_id и traffic_limit_mb
    new_expires_at = existing_expires_at
```

## Проблемы текущей логики

### Проблема 1: Сложность

**28+ проверок** для определения покупки vs продления:
- Проверка наличия подписки
- Проверка бесплатной подписки
- Проверка возраста подписки (2 порога: 1 час и 30 минут)
- Проверка соответствия expires_at ожидаемому
- Проверка других completed платежей
- Проверка флага purchase_notification_sent
- И т.д.

### Проблема 2: Возможность ошибок

**Пример (подписка #272):**
- Подписка была создана на 90 дней вместо 30
- Возможно, из-за неправильного определения покупки vs продления
- Или из-за race condition при создании

### Проблема 3: Неоднозначность

**Граничные случаи:**
- Подписка создана 59 минут назад - это покупка или продление?
- Подписка создана 31 минуту назад, но уведомление отправлено - это покупка или продление?
- Есть другие платежи, но они не связаны с подпиской (нет subscription_id) - как определить?

### Проблема 4: Зависимость от флагов

**Флаг `purchase_notification_sent`:**
- Используется для определения покупки vs продления
- Но это флаг уведомления, а не бизнес-логики
- Если уведомление не отправилось, флаг может быть не установлен, что приведет к неправильному определению

## Примеры работы логики

### Пример 1: Первый платеж

**Сценарий:**
- Пользователь оплатил первый платеж
- Подписки нет

**Результат:**
- `existing_subscription_row = None`
- → **ПОКУПКА**
- Создается подписка: `expires_at = now + 30 дней`
- Отправляется уведомление о покупке

### Пример 2: Второй платеж (нормальный)

**Сценарий:**
- Пользователь оплатил второй платеж
- Подписка существует, создана 2 дня назад
- `purchase_notification_sent = 1`
- Есть другие completed платежи

**Результат:**
- `existing_subscription_row != None`
- `subscription_age = 2 дня > 1 час` → не очень недавно
- → **ПРОДЛЕНИЕ**
- Подписка продлевается: `expires_at = expires_at + 30 дней`
- Отправляется уведомление о продлении

### Пример 3: Retry первого платежа

**Сценарий:**
- Пользователь оплатил первый платеж
- Подписка создана 10 минут назад
- `purchase_notification_sent = 0`
- Нет других completed платежей
- Платеж обрабатывается повторно (retry)

**Результат:**
- `existing_subscription_row != None`
- `subscription_age = 10 минут < 30 минут` → недавно
- `purchase_notification_not_sent = True`
- `other_completed_count = 0`
- → **ПОКУПКА** (retry)
- Подписка НЕ продлевается (уже создана с правильным сроком)
- Отправляется уведомление о покупке

### Пример 4: Проблемный случай (подписка #272)

**Сценарий:**
- Первый платеж: 2026-01-15 13:29:45
- Подписка создана: 2026-01-15 13:15:52 (за 14 минут до платежа)
- Подписка создана на 90 дней вместо 30

**Возможные причины:**
1. Подписка была создана автоматически до платежа (возможно, бесплатная)
2. При обработке первого платежа система определила это как продление вместо покупки
3. Или произошла ошибка при создании подписки (неправильный расчет expires_at)

**Проблема:** Логика не учитывает случай, когда подписка создана ДО платежа.

## Выводы

Текущая логика определения покупки vs продления:

1. **Слишком сложная** - 28+ проверок
2. **Зависит от множества факторов** - возраст подписки, флаги, другие платежи
3. **Имеет граничные случаи** - может неправильно определить тип операции
4. **Трудно отлаживать** - множество условных ветвлений
5. **Склонна к ошибкам** - как показал пример подписки #272

**Именно поэтому предлагается упрощение** - всегда пересчитывать `expires_at` на основе всех платежей, без разделения на покупку и продление.
