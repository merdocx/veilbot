# Анализ: Когда может произойти retry webhook после статуса `paid`?

## Когда YooKassa отправляет retry webhook?

**YooKassa отправляет retry webhook в следующих случаях:**

1. **Webhook не ответил 200 OK:**
   - Сервер вернул ошибку (4xx, 5xx)
   - Таймаут (не ответил в течение определенного времени)
   - Сетевые проблемы (запрос не дошел до сервера)

2. **Webhook обработался, но вернул ошибку:**
   - Обработчик вернул `False` или бросил exception
   - YooKassa считает, что webhook не обработан успешно

3. **YooKassa повторяет webhook автоматически:**
   - YooKassa может отправить один и тот же webhook несколько раз
   - Это гарантирует доставку уведомления

## Когда платеж остается в статусе `paid` (не `completed`)?

**Платеж остается в статусе `paid` если:**

1. **Обработка подписки упала с ошибкой** (строки 104-112 в `webhook_service.py`):
   ```python
   success, error_msg = await subscription_service.process_subscription_purchase(payment_id)
   
   if success:
       logger.info(f"Subscription purchase processed successfully...")
   else:
       # Если обработка не удалась, логируем ошибку, но не блокируем webhook
       # Платеж уже помечен как paid, обработка будет повторена при следующем webhook'е
       logger.error(f"CRITICAL: Failed to process subscription purchase... Will retry on next webhook")
       # НЕ возвращаем False, т.к. платеж уже помечен как paid
       # YooKassa может повторно отправить webhook, и мы попробуем обработать снова
   ```

   **Проблема:** Если обработка подписки упала, платеж остается в статусе `paid`, но подписка уже создана!

2. **Обработка подписки успешна, но обновление статуса `completed` не произошло:**
   - Обработка подписки прошла успешно
   - Но обновление статуса `completed` в БД не произошло (ошибка БД, транзакция откатилась)
   - Платеж остается в статусе `paid`

3. **Race condition между процессами:**
   - Процесс 1: Обрабатывает подписку → создает подписку → пытается обновить статус на `completed`
   - Процесс 2: Одновременно обрабатывает retry webhook → видит статус еще `paid` → пытается обработать снова

## Сценарий retry первого платежа (критичный)

**Детальный сценарий:**

1. **Первая обработка webhook:**
   ```
   Webhook 1 приходит → статус `paid` (обновлен)
   → process_subscription_purchase() вызывается
   → Проверка: подписки нет → ПОКУПКА
   → Создается подписка: expires_at = now + 30 дней
   → Платеж обновляется: subscription_id = 272
   → Статус обновляется: completed
   → Уведомление отправлено
   ```

2. **Retry webhook (через несколько секунд/минут):**
   ```
   Webhook 2 приходит (retry от YooKassa)
   → Проверка статуса в handle_yookassa_webhook (строка 91):
     if payment.status == PaymentStatus.COMPLETED:
         return True  # Пропускаем
   → Если статус еще `paid` (не обновлен) или есть race condition:
   → process_subscription_purchase() вызывается снова
   → Проверка в начале функции (строка 89):
     if payment.status == PaymentStatus.COMPLETED:
         return True  # Пропускаем
   → Если статус еще `paid`:
   → Проверка: подписка УЖЕ СУЩЕСТВУЕТ → ПРОДЛЕНИЕ
   → Подписка продлевается: expires_at = expires_at + 30 дней = 60 дней вместо 30
   ```

**Когда это может произойти?**

1. **Ошибка при обновлении статуса `completed`:**
   - Обработка подписки успешна
   - Но обновление статуса `completed` упало с ошибкой
   - Платеж остается в статусе `paid`
   - Retry webhook видит статус `paid` → обрабатывает повторно

2. **Race condition:**
   - Процесс 1: Создает подписку, обновляет `subscription_id`, начинает обновлять статус на `completed`
   - Процесс 2 (retry webhook): Проверяет статус → еще `paid` → обрабатывает → видит подписку → продлевает

3. **Транзакция не закоммичена:**
   - Обработка в транзакции, но коммит еще не произошел
   - Retry webhook видит статус `paid` (до коммита) → обрабатывает

## Вывод: Нужна ли проверка `subscription_id`?

**Да, нужна!** Проверка `subscription_id` решает проблему:

```python
# В начале process_subscription_purchase, после проверки статуса
if payment.subscription_id is not None:
    # Платеж уже обработан и связан с подпиской
    # Это retry webhook или параллельная обработка
    # Не продлевать, только отправить уведомление (если не отправлено)
    logger.info(f"Payment {payment_id} already linked to subscription {payment.subscription_id}, skipping processing")
    return send_notification_only_if_needed(payment)
```

**Почему это работает:**
- `subscription_id` обновляется **раньше**, чем статус `completed`
- Даже если статус еще `paid`, `subscription_id` уже установлен
- Это явный признак, что платеж уже обработан

## Итоговый ответ: Зачем retry после `paid`?

**Retry после `paid` может произойти из-за:**

1. **Ошибка обработки подписки:**
   - Подписка создана, но обработка упала
   - Платеж остался в статусе `paid` (не обновлен на `completed`)
   - YooKassa отправляет retry webhook

2. **Ошибка обновления статуса `completed`:**
   - Подписка создана и обработана
   - Но обновление статуса `completed` упало
   - Платеж остается в статусе `paid`
   - Retry webhook видит `paid` → обрабатывает повторно

3. **YooKassa автоматически отправляет retry:**
   - Даже если webhook обработался успешно
   - YooKassa может отправить retry для гарантии доставки
   - Это стандартное поведение платежных систем

4. **Race condition:**
   - Параллельная обработка одного платежа
   - Один процесс создал подписку, второй видит `paid` и обрабатывает повторно

**Решение:** Проверка `subscription_id` решает все эти случаи, так как `subscription_id` обновляется раньше, чем статус `completed`.
