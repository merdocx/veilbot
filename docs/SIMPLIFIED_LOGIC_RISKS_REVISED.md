# Анализ рисков упрощенной логики (с учетом того, что подписка создается только при webhook success)

## Уточнение

**Важно:** Подписка создается **только** при получении webhook `success` payment. Подписки не создаются заранее (кроме бесплатных при регистрации).

## Реальные риски упрощенной логики

### ✅ Риск 1: Webhook retry первого платежа (КРИТИЧНО)

**Сценарий:**
1. **Первая обработка webhook:**
   - Платеж в статусе `paid`
   - Проверка: подписки нет → **ПОКУПКА**
   - Создается подписка: `expires_at = now + 30 дней`
   - Платеж помечается как `completed`
   - Уведомление отправлено

2. **Webhook retry (через несколько секунд/минут):**
   - Платеж **еще в статусе `paid`** (обновление статуса не дошло до YooKassa или не было сохранено)
   - Проверка: **подписка уже существует** → **ПРОДЛЕНИЕ**
   - Подписка продлевается: `expires_at = expires_at + 30 дней` = 60 дней вместо 30

**Проблема:**
- Первый платеж создает подписку на 30 дней
- Retry продлевает её еще на 30 дней = **60 дней вместо 30**
- Пользователь получает лишние 30 дней бесплатно

**Текущая защита:**
- Проверка статуса `completed` в начале функции (строки 89-91):
  ```python
  if payment.status == PaymentStatus.COMPLETED:
      logger.info(f"Payment {payment_id} already completed, skipping")
      return True, None
  ```

**Надежность защиты:**
- ⚠️ **Зависит от атомарности обновления статуса `completed`**
- ⚠️ Если два процесса одновременно проверяют статус `paid` и оба пытаются обработать
- ⚠️ Один создает подписку, второй видит подписку и продлевает её

**Вывод:** ⚠️ **КРИТИЧЕСКИЙ РИСК** - нужна дополнительная защита

**Решение:**
1. ✅ **Проверка `subscription_id` в платеже:**
   ```python
   if payment.subscription_id is not None:
       # Платеж уже связан с подпиской
       # Не продлевать, только отправить уведомление (если не отправлено)
       return send_notification_only()
   ```

2. ✅ **Атомарное обновление статуса `completed`:**
   ```python
   # Используем try_update_status для атомарности
   update_success = await payment_repo.try_update_status(
       payment_id,
       PaymentStatus.COMPLETED,
       PaymentStatus.PAID
   )
   if not update_success:
       # Статус уже изменился другим процессом → пропускаем
       return True, None
   ```

### ✅ Риск 2: Параллельные платежи (средний)

**Сценарий:**
1. Пользователь оплатил 2 платежа одновременно (или почти одновременно)
2. Оба webhook'а приходят одновременно
3. Оба платежа в статусе `paid`
4. Оба процесса обрабатываются параллельно

**Сценарий 2A: Оба платежа - первый для пользователя**
- Процесс A: Проверяет подписку → нет → создает подписку → `expires_at = now + 30 дней`
- Процесс B: Проверяет подписку → видит подписку (созданную процессом A) → **ПРОДЛЕНИЕ** → `expires_at = expires_at + 30 дней` = 60 дней

**Результат:**
- ✅ Правильно: 2 платежа по 30 дней = 60 дней

**Текущая защита:**
- Атомарное создание подписки в транзакции (`BEGIN IMMEDIATE`)
- Если подписка создана другим процессом, второй процесс использует существующую
- Атомарное обновление `expires_at` через SQL (`extend_subscription_by_duration_async`)

**В упрощенной логике:**
- ✅ Та же защита (атомарное создание и обновление)
- ✅ Результат правильный: 60 дней

**Вывод:** ✅ **НЕТ ПРОБЛЕМЫ** - атомарные операции защищают от race conditions

**Сценарий 2B: Один платеж первый, второй - продление существующей подписки**
- Платеж 1: Создает подписку → `expires_at = now + 30 дней`
- Платеж 2: Видит подписку → продлевает → `expires_at = expires_at + 30 дней` = 60 дней

**Результат:**
- ✅ Правильно: 2 платежа по 30 дней = 60 дней

**Вывод:** ✅ **НЕТ ПРОБЛЕМЫ**

## Рекомендации по защите

### 1. Обязательная защита от retry (критично)

**Вариант A: Проверка `subscription_id` в платеже (рекомендуется)**

```python
async def process_subscription_payment(payment_id: str):
    # Шаг 1: Получаем платеж
    payment = await payment_repo.get_by_payment_id(payment_id)
    
    # Шаг 2: Проверяем статус
    if payment.status == PaymentStatus.COMPLETED:
        return True, None  # Уже обработан
    
    # Шаг 3: КРИТИЧНО - проверяем subscription_id
    if payment.subscription_id is not None:
        # Платеж уже связан с подпиской → это retry
        # Не продлевать, только отправить уведомление (если не отправлено)
        logger.info(f"Payment {payment_id} already linked to subscription {payment.subscription_id}, skipping processing")
        return send_notification_only_if_needed(payment)
    
    # Шаг 4: Определяем покупка/продление
    existing_subscription = get_active_subscription(payment.user_id)
    
    if existing_subscription:
        # ПРОДЛЕНИЕ
        return extend_subscription(...)
    else:
        # ПОКУПКА
        return create_subscription(...)
```

**Преимущества:**
- ✅ Простая и надежная проверка
- ✅ Явно показывает, что платеж уже обработан
- ✅ Не зависит от статуса (который может быть не атомарно обновлен)

---

**Вариант B: Атомарное обновление статуса `completed`**

```python
# В начале функции, после получения платежа
update_success = await payment_repo.try_update_status(
    payment_id,
    PaymentStatus.COMPLETED,
    PaymentStatus.PAID
)

if not update_success:
    # Статус уже изменился другим процессом → пропускаем
    payment_check = await payment_repo.get_by_payment_id(payment_id)
    if payment_check.status == PaymentStatus.COMPLETED:
        return True, None  # Уже обработан
```

**Недостатки:**
- ⚠️ Зависит от надежности `try_update_status`
- ⚠️ Может быть race condition между проверкой и обновлением

**Преимущества:**
- ✅ Уже реализовано в коде
- ✅ Проверяется в начале функции

### 2. Рекомендуемая комбинация защиты

**Двухуровневая защита:**

```python
# Уровень 1: Проверка статуса (быстрая проверка)
if payment.status == PaymentStatus.COMPLETED:
    return True, None

# Уровень 2: Проверка subscription_id (надежная проверка)
if payment.subscription_id is not None:
    # Платеж уже обработан
    return send_notification_only_if_needed(payment)

# Уровень 3: Атомарное обновление статуса перед обработкой
update_success = await payment_repo.try_update_status(
    payment_id,
    PaymentStatus.COMPLETED,
    PaymentStatus.PAID
)

if not update_success:
    # Другой процесс обработал платеж → пропускаем
    return True, None
```

**Вывод:** ✅ **КРИТИЧНО ДОБАВИТЬ** проверку `subscription_id` для защиты от retry

## Финальная оценка надежности упрощенной логики

### ✅ Упрощенная логика надежна, ЕСЛИ:

1. ✅ **Добавлена проверка `subscription_id`:**
   - Если `payment.subscription_id != None` → не продлевать (retry)
   - Или использовать атомарное обновление статуса `completed`

2. ✅ **Используется пересчет вместо продления:**
   - Не `expires_at = expires_at + duration`
   - А пересчет: `expires_at = first_payment + sum(all_payments_duration)`
   - Это решает проблему накопления ошибок

3. ✅ **Сохранены атомарные операции:**
   - Создание подписки в транзакции (есть)
   - Обновление `expires_at` через SQL (есть)

### ❌ Упрощенная логика НЕ надежна, ЕСЛИ:

1. ❌ **Нет проверки `subscription_id`:**
   - Webhook retry может привести к двойному продлению
   - Решение: добавить проверку `subscription_id`

2. ❌ **Используется продление вместо пересчета:**

   - Решение: пересчет на основе всех платежей
